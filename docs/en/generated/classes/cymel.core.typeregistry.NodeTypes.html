class cymel.core.typeregistry.NodeTypes¶

class cymel.core.typeregistry.NodeTypes(*args, **kwargs)¶

Bases: object

Manager for wrapper classes for node types.

The only instance, nodetypes, has been created.

All node classes can access this attribute.

Cymel provides wrapper classes for all node types, including plugins, but other than the main node types that are provided in advance for functional implementation, they are automatically generated when needed, such as when accessed for the first time.

The standard class name provided by the system is the node type name with the first letter capitalized.

You can also create a custom class by inheriting an existing node class. A custom class can be determined purely by the node type, or it can be determined by some other arbitrary condition.

Custom classes can be used as is without being registered in the system, but they can also be registered using registerNodeClass. When obtaining an instance, you can obtain the instance by specifying the class directly, but if the class is registered, you can have the class automatically determined by obtaining the instance from CyObject.

Note

It supports name specification that maintains compatibility between versions, such as changing the node type name in 2026, and it is recommended to always specify a new type name (for example, a name with the suffix DL even before 2026).

registerNodeClass(cls, nodetype)¶

Register a node class.

Node classes are divided into classes with a verification method, which have a static method _verifyNode for checking conformance, and basic classes, which do not have it.

The specifications of the static method for compliance testing are as follows.

  • _verifyNode(mfn, name)

    Assume that an instance of the MFnDependencyNode derived class of Python API 2 (for example, MFnDagNode for dagNode) and a unique name such as a partial path of the node are passed as arguments, and a Boolean value indicating suitability is returned.

When implementing _verifyNode, it is recommended to also override Node.createNode so that the conditions are met when creating a node.

Basic classes are strictly linked to node types, and either one exists or is automatically generated for every node type, including abstract types. Even when registering a custom class as a basic class, it must inherit from a parent class that does not conflict with the node type. In a class implementation, parentBasicNodeClass is useful for getting the parent class to inherit from. Conflicting registrations for one node type will result in overwriting registrations, and a warning will be output and all registrations of the old class, including subclasses, will be deleted (there is no harm in overwriting the automatic registrations that the system previously made).

On the other hand, classes with conformance checking methods do not need to be strictly linked to node types. As long as there is no conflict between class inheritance and the registered node type, you are free to register it as any node type, including abstract types. There is no problem with multiple classes with inspection methods being registered in conflict with one node type, and the same class may be registered with multiple node types. When registered, it is inserted at the beginning of the list managed for each node type, so those registered later have priority.

Parameters:

  • cls (type) – Class to register for.

  • nodetype (str) – Node type name to link.
